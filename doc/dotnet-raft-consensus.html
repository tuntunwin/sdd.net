01

What Goes Through Raft vs What Stays Local

Raft is a consensus protocol for ordering writes, not a general message bus. The critical insight: only the desired-state mutations go through Raft. The QoS delivery outbox, session management, and fire-and-forget commands stay local to each node.

▶ THROUGH RAFT (replicated, linearizable)
📦
Retained state writes (SET) Any PUBLISH with retain:true. This is a desired-state mutation. Every node must agree on the latest payload for each topic. Raft ensures all nodes apply it in the same order.
🗑
Retained state deletes Publishing a zero-length payload to a retained topic clears it. Must also go through Raft so all nodes delete in the same log position.
📋
Subscription persistence (optional) If you want "durable subscriptions" — where a clientId's topic subscriptions survive broker restart and are visible across nodes — they must be Raft-replicated. Off by default.
▷ LOCAL ONLY (per-node, not replicated)
💨
QoS 0 commands (fire-and-forget) Ephemeral by definition. No storage, no Raft. Delivered on the node the publisher connects to. If subscriber is on another node, use broker bridge or client load balancing.
📬
QoS 1 delivery outbox The per-clientId pending delivery tracking is local. Each node manages its own connected clients. If a client reconnects to a different node, it re-subscribes and receives current retained state from Raft-consistent StateStore.
🔌
Session / connection state Which clientIds are connected to which node is local. Raft does not replicate live connection maps — this would create enormous churn for no benefit.
💓
Heartbeats and pings Client keepalive PING/PONG is purely between client and its connected broker node.
Core Invariant After any committed Raft write, every live node's StateStore.Get(topic) returns the same payload — regardless of which node the client queries. This is the single-source-of-truth guarantee. The QoS delivery machinery on top of it is local per-node, but it always starts from this consistent foundation when pushing retained state to new subscribers.
02

Three-Node Architecture

Minimum 3 nodes. Quorum = 2. Can lose any 1 node and continue. Writes only accepted by the Raft leader. Followers serve reads and redirect writes.

LEADER
Node 1
Broker A
Accepts all PUBLISH (retain:true). Proposes entry to Raft log. Waits for quorum ACK before committing. Applies to local StateStore. Fans out to local subscribers.
AppendEntries RPC
FOLLOWER
Node 2
Broker B
Receives log entries from leader. Votes on proposals. Applies committed entries to local StateStore in the same order. Serves retained-state reads. Redirects writes to leader.
AppendEntries RPC
FOLLOWER
Node 3
Broker C
Same as Node 2. Forms quorum with any one other node. Eligible to become leader on election timeout if Node A fails. All three maintain identical StateStore after commit.

Component Ownership

RaftConsensusLayer
Raft · writes leader-only
Implements IConsensusLayer. Wraps IRaftCluster.ReplicateAsync(). On non-leader nodes, throws LeaderChangedException which triggers client redirect response.
BrokerStateMachine
Raft · applied on all nodes
Implements IRaftStateMachine. Raft calls ApplyAsync() only after quorum commits. Deserializes log entry → calls StateStore.SetMemory() (no WAL — Raft log IS the WAL).
BrokerPersistentState
Raft · log + snapshot storage
Extends PersistentState (DotNext). Stores Raft log on disk. Implements CreateSnapshotAsync to serialize full StateStore. Implements ApplySnapshotAsync for new node catch-up.
StateStore (sharded)
shared · read any node
Same 64-shard RWLS design. In Raft mode, Set() is called only from BrokerStateMachine.ApplyAsync() — not directly from the publish handler. Reads serve any node without Raft round-trip.
LeaderRouter
cluster · redirect writes
Intercepts PUBLISH frames on followers. Returns { type: "REDIRECT", leader: "broker-a:8080" } to client. Client SDK auto-reconnects to leader. Eliminates proxy hop overhead.
QoS Engine (local)
local · not replicated
Per-node, per-clientId outbox. Not Raft-replicated. On reconnect to any node, client re-subscribes and receives latest retained state from the consistent StateStore.

Raft Write Path — Step by Step

A retained-state PUBLISH touching the Raft layer. The key change vs single-node: the StateStore is not updated until Raft commits. The state machine's ApplyAsync is the only write gate.

Phase 1 — Publisher side
1
Client connects to any node → sends PUBLISH retain:true, QoS 1
Client may connect to a follower. The node checks consensusLayer.IsLeader. If not leader, it does NOT buffer or proxy the message.
all nodes
2
Non-leader → return REDIRECT frame to client
{ type: "REDIRECT", leader: "10.0.0.1:8080", term: 3 }. Client SDK reconnects to leader address. No message loss — client hasn't received ACK yet. QoS 1 guarantees the retry.
follower
Phase 2 — Leader processes write
3
Leader: encode WALEntry → propose to Raft log
Serialize { topic, payload, ts } to ReadOnlyMemory<byte>. Call await raftCluster.ReplicateAsync(new BrokerLogEntry(data), ct). This blocks until quorum or timeout.
leader
Phase 3 — Raft consensus (network round-trip)
4
AppendEntries RPC → followers persist to their Raft log
DotNext sends the entry to all followers. Each follower appends to its PersistentState log on disk and replies with AppendEntriesResult{success:true, term:N}.
network
5
Quorum reached (2/3 ACKs) → entry committed in Raft log
ReplicateAsync() returns on the leader. The entry's index is now the commitIndex. All nodes will eventually apply entries up to this index.
committed
Phase 4 — State machine apply (all nodes)
6
BrokerStateMachine.ApplyAsync() called on every node
DotNext calls ApplyAsync() with committed entries in order. Deserialize entry → StateStore.SetMemory(topic, payload). No WAL here — the Raft log IS the write-ahead log. Memory-only update, fast path.
all nodes
Phase 5 — Delivery (leader only for this publish)
7
Leader fans out DELIVER to locally connected subscribers
After Apply completes, the leader (which handled the original PUBLISH) fans out to its local subscriber set via the QoS Engine. Subscribers on other nodes receive the state when they query via SUBSCRIBE or reconnect.
leader-local
8
Leader sends PUBACK to publisher — write is complete
Only now does the publisher receive its QoS 1 ACK. The retained state is durable across a quorum of nodes. Publisher's at-least-once delivery guarantee is fully satisfied.
durable
No Double WAL When Raft is enabled, IWalWriter is injected as NullWalWriter. The Raft PersistentState on each node is the WAL. Writing to both would duplicate durability work for no benefit. The StateStore.SetMemory() call in ApplyAsync skips the WAL entirely — it only updates the in-memory shard map.
04

The State Machine — Core of Raft Integration

The state machine is what connects the Raft log to the actual StateStore. DotNext calls ApplyAsync() on every node after quorum commits an entry. It must be deterministic and idempotent — same input always produces the same state transition.

C# — BrokerLogEntry + BrokerStateMachineusing DotNext.Net.Cluster.Consensus.Raft;
using DotNext.IO;

// ── Log entry: what gets written into the Raft log ─────────────────────
public readonly struct BrokerLogEntry : IRaftLogEntry
{
    private readonly ReadOnlyMemory<byte> _data;   // serialized StateEntry

    public BrokerLogEntry(ReadOnlyMemory<byte> data) => _data = data;

    public long      Term      { get; init; }
    public DateTimeOffset Timestamp => DateTimeOffset.UtcNow;
    public bool      IsSnapshot => false;
    public int?      CommandId  => null;

    public ValueTask WriteToAsync<TWriter>(TWriter writer, CancellationToken ct)
        where TWriter : IAsyncBinaryWriter
        => writer.WriteAsync(_data, ct);
}

// What each log entry carries
public sealed record StateEntry(
    string       Topic,
    JsonElement  Payload,
    StateOp     Op         // Set | Delete
);

public enum StateOp { Set, Delete }

// ── The state machine — called on EVERY node after commit ───────────────
public sealed class BrokerStateMachine : PersistentState
{
    private readonly IStateStore _store;
    private readonly ISubscriberNotifier _notifier; // fan-out after apply

    public BrokerStateMachine(string dataDir, IStateStore store, ISubscriberNotifier notifier)
        : base(dataDir, new Options { MaxLogEntrySize = 1024 * 64 })
    {
        _store    = store;
        _notifier = notifier;
    }

    // Called by DotNext after quorum commits the entry
    // MUST be deterministic — no timestamps, no random values here
    protected override async ValueTask ApplyAsync(LogEntry entry)
    {
        if (entry.IsSnapshot) { await RestoreSnapshotAsync(entry); return; }

        var data = await entry.ToMemoryAsync();
        var op   = JsonSerializer.Deserialize(data.Span, BrokerJsonContext.Default.StateEntry)!;

        switch (op.Op)
        {
            case StateOp.Set:
                // SetMemory skips WAL — Raft log is the WAL
                _store.SetMemory(op.Topic, op.Payload);
                break;
            case StateOp.Delete:
                _store.DeleteMemory(op.Topic);
                break;
        }

        // Notify local subscribers AFTER state is consistent
        await _notifier.NotifyAsync(op.Topic, op.Payload, op.Op);
    }

    // ── Snapshot for log compaction ───────────────────────────────────────
    protected override async ValueTask CreateSnapshotAsync(ISnapshotWriter snapshot)
    {
        var buf = new ArrayBufferWriter<byte>();
        var writer = new Utf8JsonWriter(buf);
        writer.WriteStartArray();
        foreach (var (topic, payload) in _store.GetAll())  // reads all 64 shards
        {
            writer.WriteStartObject();
            writer.WriteString("topic", topic);
            writer.WritePropertyName("payload"); payload.WriteTo(writer);
            writer.WriteEndObject();
        }
        writer.WriteEndArray();
        await writer.FlushAsync();
        await snapshot.WriteAsync(buf.WrittenMemory);
    }

    private async ValueTask RestoreSnapshotAsync(LogEntry snapshot)
    {
        var data    = await snapshot.ToMemoryAsync();
        var entries = JsonSerializer.Deserialize(data.Span, BrokerJsonContext.Default.StateEntryArray)!;
        _store.LoadBulk(entries);   // replaces all shard contents atomically
    }
}
Determinism Rule ApplyAsync receives the same bytes on every node and must produce the same state mutation. Never use DateTime.Now, Guid.NewGuid(), or any non-deterministic source inside Apply. If timestamps are needed in the payload, they must be encoded by the publisher before entering the Raft log.

Snapshots & Log Compaction

Without snapshots, every Raft log grows forever and new nodes must replay the entire history. PersistentState handles compaction automatically — it calls CreateSnapshotAsync when the log exceeds a configurable threshold.

C# — Snapshot Lifecycle// ── New node joins the cluster ──────────────────────────────────────────
//
//  Leader detects follower is far behind (commitIndex - follower.nextIndex > threshold)
//  Leader calls CreateSnapshotAsync() → serializes full StateStore
//  Leader sends snapshot via InstallSnapshot RPC
//  Follower calls RestoreSnapshotAsync() → rebuilds StateStore from snapshot
//  Follower replays only log entries AFTER snapshot's lastIncludedIndex
//
// DotNext handles the RPC mechanics. You only implement Create + Restore.

// ── Manual snapshot trigger (optional, e.g. on schedule) ───────────────
public sealed class SnapshotScheduler : BackgroundService
{
    private readonly BrokerStateMachine _sm;
    private readonly TimeSpan _interval;

    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        using var timer = new PeriodicTimer(_interval);
        while (await timer.WaitForNextTickAsync(ct))
        {
            // ForceCompactionAsync triggers CreateSnapshotAsync + log truncation
            await _sm.ForceCompactionAsync(0L, ct);  // 0 = compact all committed
        }
    }
}

Read Distribution

Writes must go to the leader. Reads have two options trading consistency for latency. Pick per request type.

C# — Read Consistency Optionspublic sealed class DistributedStateStore
{
    private readonly IStateStore  _local;
    private readonly IRaftCluster _raft;

    // ── Option A: Stale read (any node) ──────────────────────────────────
    // Returns value from local in-memory StateStore.
    // May be behind by up to one AppendEntries round-trip (~1-5ms on LAN).
    // Use for: dashboards, metrics, non-critical state polling.
    public JsonElement? GetStale(string topic)
        => _local.Get(topic);   // purely local, no network

    // ── Option B: Linearizable read (leader only) ─────────────────────────
    // Calls ReadIndex: leader confirms it's still leader by checking
    // heartbeat quorum, THEN reads local store.
    // Guarantees read reflects all prior committed writes.
    // Use for: new subscriber's initial desired-state push.
    public async ValueTask<JsonElement?> GetLinearizableAsync(
        string topic, CancellationToken ct)
    {
        // ReadIndex ensures leader hasn't been superseded since last heartbeat
        await _raft.ReadIndexAsync(ct);         // ~0.5 RTT cost
        return _local.Get(topic);              // now safe to read locally
    }

    // ── Which to use for new SUBSCRIBE ───────────────────────────────────
    // A new subscriber needs the current desired state pushed at QoS 1.
    // Use linearizable read so they don't get a stale payload that
    // was already superseded by a committed write on another node.
    public async ValueTask<IReadOnlyList<RetainedEntry>> GetRetainedForSubscribeAsync(
        string pattern, CancellationToken ct)
    {
        await _raft.ReadIndexAsync(ct);   // linearize before reading
        return _local.GetMatching(pattern);
    }
}
Follower Reads Without ReadIndex If a follower serves a new subscriber's retained state without a ReadIndex check, the subscriber might receive a stale payload — one that a committed Raft write has already superseded on the leader. For this reason, new subscriber retained-state pushes always use linearizable reads, even if the follower adds a small latency cost. Ongoing QoS 1 delivery after that initial push can be stale-safe since the subscriber will naturally converge.

RaftConsensusLayer — Full Implementation

C# — RaftConsensusLayer wrapping DotNextpublic sealed class RaftConsensusLayer : IConsensusLayer
{
    private readonly IRaftCluster         _cluster;
    private readonly BrokerStateMachine   _sm;

    public bool IsLeader => _cluster.LeadershipToken.IsCancellationRequested == false
                         && _cluster.Leader?.Endpoint.Equals(_cluster.LocalMemberAddress) == true;

    public async ValueTask ProposeAsync(ReadOnlyMemory<byte> data, CancellationToken ct)
    {
        if (!IsLeader)
            throw new NotLeaderException(_cluster.Leader?.Endpoint);

        var entry = new BrokerLogEntry(data) { Term = _cluster.Term };
        await _cluster.ReplicateAsync(entry, ct);
        // Returns only after quorum. ApplyAsync called on all nodes next.
    }
}

// ── Broker publish handler with Raft ────────────────────────────────────
public async ValueTask HandlePublishAsync(Frame frame, string senderClientId, CancellationToken ct)
{
    // QoS 0 command: bypass Raft entirely
    if (frame.Qos == 0)
    {
        var targets = _trie.Match(frame.Topic!);
        foreach (var id in targets) _sessions.TrySend(id, frame);
        return;
    }

    // Non-leader: send redirect, do not buffer
    if (!_consensus.IsLeader)
    {
        var redirect = new Frame("REDIRECT",
            Payload: JsonSerializer.SerializeToElement(
                new { leader = _consensus.GetLeaderEndpoint() },
                BrokerJsonContext.Default.LeaderInfo));
        _sessions.TrySend(senderClientId, redirect);
        return;
    }

    // Leader: propose to Raft
    if (frame.Retain)
    {
        var entry = new StateEntry(frame.Topic!, frame.Payload, StateOp.Set);
        var data  = JsonSerializer.SerializeToUtf8Bytes(entry, BrokerJsonContext.Default.StateEntry);
        await _consensus.ProposeAsync(data, ct);
        // ApplyAsync fires on all nodes → StateStore updated → NotifyAsync fans out
    }
    else
    {
        // Non-retained QoS 1: skip Raft, fan out directly with outbox
        var targets = _trie.Match(frame.Topic!);
        foreach (var id in targets) await _qos.EnqueueAsync(id, frame, ct);
    }

    // PUBACK to publisher (after Raft commit)
    _sessions.TrySend(senderClientId, new Frame("ACK", MsgId: frame.MsgId));
}

DI Registration — Wiring It Together

C# — Program.cs / Composition Rootvar cfg = BrokerConfig.Load("appsettings.json");

// Core components (always present)
var store    = new ShardedStateStore(cfg.Store.Shards);
var sessions = new SessionManager(cfg.Session.SendBufferDepth);
var qos      = new QosEngine(cfg.Qos, sessions);
var trie     = new TopicTrie();
var notifier = new SubscriberNotifier(trie, sessions, qos);

IConsensusLayer consensus;
IWalWriter      wal;

if (cfg.Cluster.Mode == "raft")
{
    // Raft mode: stateMachine is both the WAL and the apply gate
    var sm = new BrokerStateMachine(cfg.Store.WalDir, store, notifier);

    var raftCluster = await RaftCluster.CreateAsync(new RaftCluster.NodeConfiguration
    {
        PublicEndPoint   = cfg.Cluster.LocalEndpoint,
        Members          = cfg.Cluster.Peers.Select(IPEndPoint.Parse).ToArray(),
        ElectionTimeout  = cfg.Cluster.RaftElectionTimeout,
        HeartbeatThreshold = 2,
    }, sm);

    await raftCluster.StartAsync();
    consensus = new RaftConsensusLayer(raftCluster, sm);
    wal       = new NullWalWriter();     // Raft IS the WAL
}
else
{
    // Single-node mode: custom WAL, passthrough consensus
    wal       = new WalWriter(cfg.Store.WalDir);
    consensus = new NoopConsensus();
}

// Inject WAL into store (Raft mode: NullWal; single: real WAL)
store.ConfigureWal(wal);

var broker = new Broker(store, sessions, qos, trie, notifier, consensus, wal);
await broker.RunAsync(cfg);

Key Rules — What Must Never Happen

RuleWhy
Never call StateStore.SetAsync() directly from publish handler in Raft mode State must only change inside ApplyAsync. Direct writes bypass quorum ordering and create split-brain.
Never await inside ApplyAsync across unrelated state DotNext calls Apply sequentially. Async awaits that depend on other state can deadlock the apply pipeline.
Never use WalWriter when Raft is enabled Double durability — Raft's PersistentState already provides the WAL. Using both wastes IOPS and can diverge.
Never serve linearizable reads from a follower without ReadIndex A stale follower may return a payload that the leader already superseded via a committed write.
Never route QoS 0 commands through Raft Fire-and-forget by definition. Adding Raft overhead would change QoS 0 semantics and add unnecessary latency.
Summary — The Clean Mental Model Think of the Raft log as a shared, ordered journal that every node subscribes to. A publisher writes a state mutation intent into the journal. Once 2 of 3 nodes have written it to their disk, it's committed. Then every node's state machine reads it from the journal and applies it. The StateStore is simply the materialized view of that journal — rebuilt from the latest snapshot plus all committed entries after it. QoS delivery (outbox, retries, reconnect replay) is the delivery mechanism on top of that consistent store, and it stays local per-node.
