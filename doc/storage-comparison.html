01

Broker Storage Access Pattern

Before comparing engines, the workload must be profiled precisely. The wrong engine for the access pattern is worse than a simple custom solution. Our broker's storage layer is not a general-purpose database — it has very specific characteristics.

~55%
Sequential WAL Append
Every retained state write appends a JSON entry to the active WAL segment. Fully sequential. The dominant I/O pattern — must be near disk throughput ceiling.
~25%
Random KV Write (StateStore)
Updating the in-memory retained state map after WAL confirms. Eventually flushed via snapshot. Topics are updated in arbitrary order — random key space.
~15%
Point Read (GET topic)
New subscriber SUBSCRIBE triggers Get(topic). Must be fast — blocks the subscriber's initial desired-state push. Hot topics may be cached warm in OS page cache.
~4%
Full Scan (restart + snapshot)
On broker restart: replay WAL into StateStore. On snapshot: iterate all 64 shards. Infrequent but must complete in seconds, not minutes, for acceptable restart time.
~1%
Delete (topic clear)
Publishing zero-length payload to a retained topic removes it. Rare. Tombstoned in the WAL. No tombstone accumulation concern at this volume.
Critical Insight The dominant write path is sequential WAL append, not random KV writes. Engines optimised for random writes (LSM-tree compaction overhead) add unnecessary complexity. The actual retained state KV store is mostly in-memory — disk is only for durability on crash. This means an engine that prioritises sequential write throughput and fast point reads from an in-memory-backed store is the best fit — not necessarily the engine with the highest benchmark random-write ops.

Write Amplification Breakdown

Write amplification (WA) is the ratio of bytes written to disk vs bytes in the original write. For our WAL-dominant broker, high WA directly translates to lower throughput and faster SSD wear. LSM-tree engines are the worst offenders due to multi-level compaction.

Why LSM-tree WA is High for Our Use Case LSM-tree engines buffer writes in a memory table (MemTable), flush to L0 SSTables, then compact through multiple levels. Each compaction reads and rewrites data. For our broker's use case — where topics are updated frequently but the total key space is modest — the compaction overhead is disproportionate. The broker doesn't need the LSM advantage (absorbing massive random write bursts) because writes flow sequentially through the WAL first anyway.

Integration Code — Swapping the Storage Backend

All engines plug in behind the same IWalWriter / IStateStore interfaces. Here are the relevant integration patterns for the strongest alternatives.

RocksDB — Go (via grocksdb)

Go — RocksDB StateStore// go get github.com/linxGnu/grocksdb
// Requires librocksdb.so on the host — NOT a single binary

import "github.com/linxGnu/grocksdb"

type RocksStateStore struct {
    db   *grocksdb.DB
    ro   *grocksdb.ReadOptions
    wo   *grocksdb.WriteOptions
}

func NewRocksStateStore(dir string) (*RocksStateStore, error) {
    opts := grocksdb.NewDefaultOptions()
    opts.SetCreateIfMissing(true)
    opts.SetCompression(grocksdb.LZ4Compression)
    opts.SetMaxWriteBufferNumber(4)
    opts.SetWriteBufferSize(64 << 20)      // 64MB memtable
    opts.SetLevel0FileNumCompactionTrigger(4)

    db, err := grocksdb.OpenDb(opts, dir)
    if err != nil { return nil, err }
    wo := grocksdb.NewDefaultWriteOptions()
    wo.SetSync(false)  // WAL handles durability — RocksDB is secondary store
    return &RocksStateStore{db: db, ro: grocksdb.NewDefaultReadOptions(), wo: wo}, nil
}

func (r *RocksStateStore) Set(topic string, payload json.RawMessage) error {
    return r.db.Put(r.wo, []byte(topic), payload)
}
func (r *RocksStateStore) Get(topic string) (json.RawMessage, bool) {
    v, err := r.db.Get(r.ro, []byte(topic))
    if err != nil || !v.Exists() { return nil, false }
    defer v.Free()
    return append(json.RawMessage{}, v.Data()...), true
}

BadgerDB — Go (pure, recommended over RocksDB for this use case)

Go — BadgerDB StateStore// go get github.com/dgraph-io/badger/v4
// Pure Go — single binary, no native deps

import badger "github.com/dgraph-io/badger/v4"

type BadgerStateStore struct { db *badger.DB }

func NewBadgerStateStore(dir string) (*BadgerStateStore, error) {
    opts := badger.DefaultOptions(dir).
        WithLogger(nil).          // suppress noise
        WithSyncWrites(false).   // WAL handles durability
        WithCompression(options.Zstd)
    db, err := badger.Open(opts)
    if err != nil { return nil, err }
    return &BadgerStateStore{db: db}, nil
}

func (b *BadgerStateStore) Set(topic string, payload json.RawMessage) error {
    return b.db.Update(func(txn *badger.Txn) error {
        return txn.Set([]byte(topic), payload)
    })
}

func (b *BadgerStateStore) GetAll() (map[string]json.RawMessage, error) {
    result := make(map[string]json.RawMessage)
    err := b.db.View(func(txn *badger.Txn) error {
        opts := badger.DefaultIteratorOptions()
        it := txn.NewIterator(opts)
        defer it.Close()
        for it.Rewind(); it.Valid(); it.Next() {
            item := it.Item()
            v, _ := item.ValueCopy(nil)
            result[string(item.Key())] = v
        }
        return nil
    })
    return result, err
}

FASTER — .NET (pure C#, recommended for .NET broker)

C# — FASTER StateStore// dotnet add package Microsoft.FASTER.Core
// Pure C# — no native dependencies, NativeAOT compatible

using FASTER.core;

public sealed class FasterStateStore : IStateStore
{
    private readonly FasterKV<SpanByte, SpanByte> _store;
    private readonly IDevice _log;

    public FasterStateStore(string dir)
    {
        _log = Devices.CreateLogDevice(Path.Combine(dir, "state.log"));
        _store = new FasterKV<SpanByte, SpanByte>(
            1L << 20,  // 1M hash buckets
            new LogSettings {
                LogDevice      = _log,
                MemorySizeBits = 25,   // 32MB in-memory tier
                PageSizeBits   = 14,   // 16KB pages
            }
        );
    }

    public ValueTask SetAsync(string topic, JsonElement payload, CancellationToken ct)
    {
        var session = _store.NewSession<SpanByte, SpanByte, Empty, SimpleSimpleFunctions<SpanByte, SpanByte>>
            (new SimpleSimpleFunctions<SpanByte, SpanByte>());
        try {
            var keyBytes   = Encoding.UTF8.GetBytes(topic);
            var valueBytes = JsonSerializer.SerializeToUtf8Bytes(payload, BrokerJsonContext.Default.JsonElement);
            unsafe {
                fixed (byte* kp = keyBytes, vp = valueBytes) {
                    var key = SpanByte.FromPinnedPointer(kp, keyBytes.Length);
                    var val = SpanByte.FromPinnedPointer(vp, valueBytes.Length);
                    session.Upsert(ref key, ref val);
                }
            }
        } finally { session.Dispose(); }
        return ValueTask.CompletedTask;
    }
}
Hybrid Design — Best of Both The highest-performance design keeps the custom WAL for durability (sequential append is unbeatable) but replaces the sharded in-memory map with FASTER or BadgerDB as the KV tier. WAL → fsync → KV.Upsert(). On restart: load KV snapshot → replay WAL tail. This gives you crash safety from the WAL and efficient range queries + compression from the KV engine — without the LSM compaction interfering with the sequential write path.

Recommendation by Scenario

Go · minimal deps · <100k topics
★ Custom WAL + Snapshots
Our design. Zero deps, purpose-built for this access pattern, single binary. No compaction overhead. WAL sequential append saturates disk write bandwidth. Sufficient for the majority of deployments.
Go · no native deps · >500k topics with compression
BadgerDB
Pure Go, Zstd compression for large payloads, VLOG reduces write amplification vs plain LSM-tree. Replace the in-memory shard map with Badger — keep the WAL or disable it (Badger has its own WAL). Single binary maintained.
Go · max write throughput · accept native dep
RocksDB via grocksdb
Only if write throughput exceeds what the custom WAL can deliver and compression matters for storage cost. The C++ dependency breaks single-binary deploy and complicates cross-platform builds. Bring in only if benchmarks prove necessity.
.NET · minimal deps · <100k topics
★ Custom WAL + Snapshots
Same rationale as Go. WalWriter + ShardedStateStore is purpose-built, NativeAOT-compatible, and zero NuGet. Ideal for the majority of deployments.
.NET · high topic cardinality · need compression
Microsoft FASTER
Pure C#, no native deps, NativeAOT-compatible. Hybrid log keeps hot topics in memory, spills to disk. The best .NET-native KV option with minimal write amplification. One NuGet package.
Read-dominated · many subscribers reading same topics
LMDB
MVCC + memory-mapped zero-copy reads give the highest concurrent read throughput of any embedded engine. If the broker serves far more reads than writes (rare state updates, many subscribers polling), LMDB's read performance justifies the native C dependency.
Raft mode enabled (any language)
PersistentState (DotNext) / etcd/raft's own storage
When Raft is enabled, the Raft log is the WAL and the snapshot mechanism replaces periodic dumps. Do not stack a separate KV engine on top of Raft storage — this creates double durability overhead with no benefit.
Avoid in all scenarios
SQLite for broker storage
SQLite is not designed for high-frequency KV updates from multiple concurrent goroutines/tasks. Write lock contention, B-tree page locking, and ACID overhead make it the worst performer for this exact workload despite being the most familiar option.
